# Historia de Usuarios - OPsCore

## Usuario Administrador
**Responsable de la administración del sistema y la gestión de accesos.**

### Historia 1: Creación y Gestión de Usuarios
Como Administrador, quiero registrar nuevos usuarios y asignarles roles y áreas específicas, para garantizar que solo el personal autorizado acceda a la plataforma.

**Criterios de Aceptación:**
- Solo el Administrador tiene acceso al formulario de **Registro de Usuario**.
- El formulario de creación debe incluir: Nombre, Apellido, Correo, Rol (Admin, Operador, Técnico, Supervisor) y Área del nuevo usuario (Mantenimiento, RRHH, etc.).
- El sistema valida el formato de correo (`@`, `.com`) y registra el usuario.
- El Administrador puede dar de baja o editar roles de usuarios existentes.
- El Administrador tiene el acceso y contrl total del sistema(Excepto los LOGS, los LOGS deben ser inalterables)

---

## Usuario Supervisor
**Responsable de la gestión y asignación de técnicos y control de calidad.**

### Historia 1: Asignación de Técnicos
Como Supervisor, quiero asignar un técnico específico a un incidente pendiente para iniciar el proceso de reparación.

**Criterios de Aceptación:**
- El Supervisor visualiza una lista de incidentes con estado **Abierto**.
- El sistema muestra solo los técnicos disponibles dentro del área del incidente.
- Al confirmar la asignación, el estado del incidente cambia a **ASIGNADO**.

### Historia 2: Validación y Cierre de Incidente
Como Supervisor, quiero revisar la solución aportada por el técnico para cerrar el incidente formalmente.

**Criterios de Aceptación:**
- El Supervisor accede a los incidentes con estado **SOLUCIONADO**.
- El sistema permite visualizar el tiempo empleado para la solución del incidente.

### Historia 3: Configuración de Áreas y Parámetros
Como Supervisor, quiero definir las áreas de la empresa y los tipos de incidentes para que la plataforma esté alineada con la estructura organizacional.

**Criterios de Aceptación:**
- El Supervisor puede crear, editar, eliminar o configurar las áreas (Ej: Mantenimiento, Sistemas).
- El Supervisor puede crear, editar, eliminar o configurar la prioridad de los incidentes (Ej: Alta, Media, Baja).
- El Supervisor puede crear, editar, eliminar o configurar los tipos de incidentes (Ej: Fallo de máquina, problema de calidad, accidente).
- El Supervisor puede crear, editar, eliminar o configurar los estados de incidentes (Ej: Abierto, Asignado, En Proceso, Solucionado, Finalizado, Reasignado).

## Usuario Operador
**Responsable del registro o reporte de incidentes.**

### Historia 1: Registro de Incidentes
Como Operador, quiero registrar un nuevo incidente en el sistema para que sea procesado y asignado a un técnico.

**Criterios de Aceptación:**
- El Operador puede completar un formulario de reporte de incidentes con los siguientes campos: título, descripción, área, tipo de incidente y área afectada.
- El sistema asigna automáticamente un ID único y el estado **Abierto**.

---

## Usuario Técnico
**Responsable de la ejecución y resolución de tareas.**

### Historia 1: Gestión de Tareas Asignadas
Como Técnico, quiero visualizar los incidentes que tengo asignados para proceder con la reparación física o lógica.

**Criterios de Aceptación:**
- El Técnico tiene una vista de **Mis Tareas** donde solo aparecen los incidentes asignados por el Supervisor.
- El sistema muestra al técnico una descripción clara de la tarea asignada.

### Historia 2: Reporte de Solución
Como Técnico, quiero marcar un incidente como **Solucionado** para que el Supervisor valide mi trabajo.

**Criterios de Aceptación:**
- El Técnico debe completar obligatoriamente un campo de **Descripción de la Solución**.
- Al confirmar el incidente como **Solucionado**, el incidente desaparece de su bandeja de tareas activas.
- Posteriormente, el incidente será validado por el Supervisor, quien determinará si lo **Finaliza** o lo **Reabre**.

## Funcionalidades Generales (Comunes)

### Historia 1: Inicio de Sesión y Seguridad
Como Usuario de OpsCore, quiero ingresar con mis credenciales para acceder a las funciones según mi rol.

**Criterios de Aceptación:**
- Validación estricta de correo y contraseña.
- La contraseña debe tener al menos 8 caracteres, una letra mayúscula, letras minúsculas, un número y un carácter especial sin espacios.
- El sistema identifica el rol del usuario y lo redirige a su vista correspondiente (Dashboard de Admin, Operador, etc.).
- El sistema mantiene la sesión segura mediante tokens JWT.

### Historia 2: Reportes y Métricas (Dashboard)
Como Usuario, quiero ver indicadores de gestión para medir el desempeño del área.

**Criterios de Aceptación:**
- El Admin/Supervisor ve métricas globales (tiempo promedio de resolución, incidentes por área).
- El Técnico ve su propia estadística de incidentes resueltos.
- El sistema muestra alertas visuales para incidentes que han superado el tiempo límite de respuesta.
